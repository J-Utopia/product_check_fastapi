# 핵심 로직 / 아키텍처

## 한 줄 구조

`GPTs -> FastAPI -> 모드투어 API들 -> 정규화 -> 검증룰 -> 점수/이슈 반환`

## 내가 관리해야 하는 파일

- `app/main.py`
  - FastAPI 진입점
  - `/`, `/health`, `/run-itinerary`를 관리
- `app/client.py`
  - 외부 API 호출
  - 인증 헤더와 요청 조합
- `app/auth.py`
  - 인증 헤더 수집/재사용
  - `.cache/modetour_headers.json`을 우선 재사용
- `app/normalizer.py`
  - 여러 API 응답을 검수용 구조로 정리
  - 제목, 포함/불포함, 항공, 일정, 호텔, 핵심포인트 추출
- `app/rules.py`
  - 실제 오류 판정
  - 점수, 등급, 이슈 목록 생성
- `app/models.py`
  - 응답 스키마 정의
- `app/service.py`
  - 전체 흐름 오케스트레이션
- `검증룰1.json` ~ `검증룰5.json`
  - GPTs와 검수 판단 기준의 원문
- `GPTs_스키마.md`, `GPTs_지침.md`
  - GPTs가 어떤 형식으로 API를 호출하고 해석할지 정의

## 무엇을 우선 봐야 하나

1. `app/main.py`
   - 서비스가 살아 있는지
   - 실제 엔드포인트가 무엇인지
2. `app/normalizer.py`
   - 어떤 값이 추출되는지
   - 텍스트가 너무 길거나 빠진 값이 없는지
3. `app/rules.py`
   - 어떤 조건을 오류로 보는지
   - 오탐이 있으면 여기부터 조정
4. `검증룰*.json`
   - GPTs 최종 판단 기준
## 실제 호출 데이터 보는 방법

### 1. 로컬 API 확인

브라우저 또는 Postman에서 아래 주소를 호출한다.

- `GET http://127.0.0.1:8010/`
- `GET http://127.0.0.1:8010/health`
- `POST http://127.0.0.1:8010/run-itinerary`

### 2. 실행 예시

```json
{
  "group_id": "105195679"
}
```

### 3. 응답에서 보는 순서

- `status`, `code`, `message`
- `meta.issue_count`, `meta.air_segment_count`, `meta.day_count`
- `result.summary`
- `result.quality`
- `result.issues`
- `result.normalized`

### 4. 중요한 값

- `title`, `product_name`
- `top_badges`
- `departure_date`, `arrival_date`, `nights`, `days`
- `city_names`, `country_names`
- `included_items`, `excluded_items`
- `special_benefits`, `sightseeings`
- `air_segments`
- `schedule_days`
- `hotels`
- `flight_remarks`

## 관리 원칙

- 실제 판단은 `summary`가 아니라 `normalized` 기준으로 한다.
- 불필요한 생성물은 지운다.
- 실행 캐시와 로그는 남기지 않는다.
- 인증 헤더 캐시는 삭제하지 않는다. 재호출 안정성에 필요하다.
- 검수 룰이 바뀌면 `rules.py`, `검증룰*.json`, GPTs 문서를 같이 맞춘다.

## 현재 남겨둔 실행 캐시

- `.cache/modetour_headers.json`

이 파일은 인증 헤더 재사용용이다.  
완전한 원본 데이터는 아니지만, 매번 수동 인증을 줄이는 데 필요하다.
