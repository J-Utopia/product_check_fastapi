# 핵심 로직 / 아키텍처

## 한 줄 구조

`GPTs -> FastAPI -> 모드투어 API -> 정규화 -> 룰엔진 -> 점수/이슈 반환`

## 핵심 파일

- `app/main.py`
  - FastAPI 진입점
  - `/health`, `/openapi.json`, `/docs`, `/run-itinerary`
- `app/auth.py`
  - 인증 헤더 우선순위 처리
  - Render 운영에서는 `MODETOUR_HEADER_CACHE_JSON` 사용 권장
- `app/client.py`
  - 모드투어 API 호출
- `app/normalizer.py`
  - 제목, 포함/불포함, 항공, 핵심포인트, 일정, 호텔, 쿠폰 정리
- `app/rules.py`
  - 제목/기간/항공/포함불포함/핵심포인트/일차/텍스트 품질 검증
- `app/service.py`
  - 전체 실행 흐름과 응답 조립
- `GPTs_스키마.md`
  - GPTs Action 등록용 스키마
- `GPTs_지침.md`
  - GPTs 판단 방식과 출력 형식
- `검증룰1.json` ~ `검증룰5.json`
  - GPTs가 참고할 검수 기준 원문

## 실제 검수 흐름

1. GPTs가 `group_id`를 받는다.
2. GPTs Action이 `/run-itinerary`를 호출한다.
3. FastAPI가 모드투어 API를 호출한다.
4. `normalizer.py`가 검수용 구조로 정리한다.
5. `rules.py`가 이슈, 점수, 등급을 계산한다.
6. GPTs가 `검증룰1~5.json`과 응답 데이터를 함께 보고 최종 설명을 만든다.

## 현재 구조에서 가능한 검수 범위

- 제목과 기간 일치 여부
- 노팁, 노쇼핑, 노옵션, 기사/가이드 경비 충돌
- 포함/불포함 논리 충돌
- 비자/무비자 문구 충돌
- 항공 편명, 직항/경유, 출발/리턴 도시, 소요시간 이상치
- 여행도시와 일정 흐름 상충 가능성
- 핵심포인트와 실제 호텔/식사 불일치
- 일차 번호, 날짜 증가, 선택관광/쇼핑 여부 충돌
- 텍스트 인코딩 이상치

## 아직 GPTs가 최종 보완해야 하는 부분

- 오탈자에 대한 자연어 수준 판단
- 문장 어투와 고객 안내 품질 평가
- 누락된 설명을 어떻게 보완할지에 대한 제안 문장 작성
- 점수에 대한 종합 총평 작성

즉, 현재 구조는 A~Z 전체 데이터를 수집하고 핵심 논리 모순을 검수하는 데 충분하다.  
최종 표현 품질과 수정 제안 정리는 GPTs가 맡는 구조가 맞다.

## 운영 체크포인트

1. `/health`만 보지 말고 `/run-itinerary`까지 확인
2. Render에는 인증 헤더 환경변수 필요
3. 점수는 API가 계산하고 GPTs는 그 점수를 해석
4. 검증룰 JSON과 `rules.py`가 크게 어긋나면 같이 수정
